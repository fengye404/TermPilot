# TermPilot 实施方案

## 1. 实施目标

这一版实施方案要服务于已经确认的产品方向：

- 会话底座以 `tmux` 为主
- 手机和电脑共享统一会话池
- 同一个会话支持双端同时查看和控制
- 智能体的流式输出要能稳定同步到手机

## 2. 开发顺序

建议按下面顺序推进：

1. 中继服务
2. PC 端 agent
3. 手机端

原因很简单：

- 中继协议先稳定下来，三端边界才清晰
- agent 跑通后，才能证明 `tmux` 会话真的能被远程观察和控制
- 手机端最后做，返工最少

## 3. 第一阶段：打通主链路

目标：

- 证明一个 `tmux` 会话可以被手机远程查看和输入

范围：

- 单用户
- 单设备
- 单会话
- 不接数据库
- 不做正式配对界面

交付物：

- 一个中继服务，支持 PC 和手机同时连入
- 一个 PC 端 agent，能够创建并管理一个 `tmux` 会话
- 一个最小客户端页面，能够查看输出并发送输入

验收标准：

- 在 PC 上通过 TermPilot 创建一个会话
- 在这个会话里执行命令后，手机能在 1 秒内看到输出
- 手机可以发送 `Enter`、`Ctrl+C` 等输入
- 断网重连后还能继续接上同一个会话

## 4. 第二阶段：做成可用原型

目标：

- 让它从“技术演示”变成“你真的可以带出门用”的工具

范围：

- 多会话
- 会话列表
- 会话创建与关闭
- 最近输出补拉
- 移动端快捷按键

### 4.1 PC 端工作

- 建立统一会话注册表
- 将每个 TermPilot 会话映射到 `tmux` 会话或窗口
- 维护每个会话的输出序号
- 支持输入、关闭、调整尺寸
- 处理断线重连

### 4.2 中继服务工作

- 按 `deviceId` 和 `sid` 路由消息
- 维护会话在线状态
- 缓存最近一段输出
- 支持重连后的按序补拉

### 4.3 手机端工作

- 会话列表页
- 会话详情页
- 终端输出渲染
- 输入框与快捷键栏
- 在线状态和重连状态提示

验收标准：

- 在手机上创建的会话，电脑端能立刻看到
- 在电脑上创建的会话，手机端能立刻看到
- 任意一端关闭会话，另一端能立刻感知
- 可以在两个以上会话之间切换
- 切换或重连后能补上最近输出

## 5. 第三阶段：做成安全可用的 MVP

目标：

- 满足个人日常使用的基本安全和稳定要求

范围：

- 用户登录
- 设备配对
- 长期设备凭证
- 基础审计日志
- 可部署的中继服务

至少要持久化的数据：

- 用户
- 设备
- 用户与设备的绑定关系
- 会话元数据

这一阶段仍然可以暂缓：

- 端到端加密
- 多用户共享
- 完整历史输出永久存储
- Redis

## 6. PC 端设计建议

PC 端应当围绕“会话后端抽象”来做，但主实现直接按 `tmux` 落地。

建议抽象一层接口，避免后面重写上层协议：

```ts
interface SessionBackend {
  createSession(input: CreateSessionInput): Promise<SessionRecord>;
  listSessions(): Promise<SessionRecord[]>;
  writeInput(sid: string, data: string): Promise<void>;
  resize(sid: string, cols: number, rows: number): Promise<void>;
  killSession(sid: string): Promise<void>;
  onOutput(listener: (event: SessionOutputEvent) => void): void;
  onExit(listener: (event: SessionExitEvent) => void): void;
}
```

但在实现上，第一优先级是：

- `TmuxBackend`

只有在需要做非常早期的技术实验时，才考虑：

- `PtyBackend`

## 7. 会话创建规则

第一版应明确规定：

- 需要跨端同步的任务，必须从 TermPilot 管理的会话里启动
- TermPilot 管理的会话，本质上要进入统一会话池
- 不在统一会话池里的终端，不保证双端同步

这条规则需要体现在产品和实现两层：

- 产品层：文案和使用方式说明清楚
- 实现层：只展示和管理已注册会话

## 8. 协议中必须尽早冻结的内容

为了减少返工，下面这些协议点应尽早定死：

- 消息外层结构
- 认证握手
- 设备标识
- 会话标识
- 输出序号规则
- 重连补拉规则
- 错误结构

建议的错误格式：

```json
{
  "type": "error",
  "reqId": "r_002",
  "code": "SESSION_NOT_FOUND",
  "message": "会话 s_123 不存在"
}
```

## 9. 手机端交互原则

手机端要解决的是“离开电脑后，随时看一眼、补一条命令、必要时关闭会话”。

所以第一版应优先优化这些动作：

- 查看最近输出
- 进入当前活跃会话
- 发送一条命令
- 中断当前任务
- 新建会话
- 关闭会话

第一版不要把重点放在这些事情上：

- 长文本输入
- 文件编辑
- 复杂多栏布局
- 过重的桌面式交互

建议的第一版体验：

- 默认进入最近会话
- 固定显示在线状态
- 底部常驻快捷键栏
- 提供 `Enter`、`Tab`、`Ctrl+C`、方向键

## 10. 部署建议

中继服务：

- 一台小型 VPS 就足够
- 用 HTTPS 和 `wss://`
- 尽量让 HTTP 和 WebSocket 同源

PC 端 agent：

- 第一版先做命令行常驻进程
- 后续再考虑 macOS 菜单栏形态
- 未来建议支持开机自动启动

手机端：

- 第一版做 PWA
- 只有在推送、后台能力明显受限时，再考虑原生壳

## 11. 测试重点

至少要覆盖下面几类验证：

- 中继服务的路由正确性
- 协议序列化与反序列化
- PC 端会话创建、输入、关闭、退出
- 输出序号连续性
- 重连后的补拉逻辑

必须手工验证的场景：

- 手机网络波动
- 智能体大量刷屏输出
- 长时间运行后再重连
- 一端关闭会话，另一端立刻感知

## 12. 当前最值得先做的事

建议下一步按下面顺序推进：

1. 先把协议外层结构定下来
2. 实现一个最小中继服务
3. 实现一个 `tmux` 版 PC 端 agent
4. 用最小网页先验证会话查看和输入
5. 确认流式输出、关闭会话、重连补拉都稳定后，再做正式手机界面
