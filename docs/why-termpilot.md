# Why TermPilot

这份文档不讲安装，也不讲命令细节。它只回答一个产品问题：

**在已经有 SSH、远程桌面和各种远控工具的前提下，TermPilot 为什么仍然值得单独存在？**

答案不是“它也能远程进终端”，而是：

**它把“继续接上原来那条还在跑的终端会话”做成了默认路径。**

## 1. 它解决的是“会话连续性”，不是“重新登录”

很多远程方案能让你重新进到一台机器里，但它们默认做的是：

- 再开一个新终端
- 再开一个新 shell
- 再建立一条新的上下文

而真实工作里，你更常遇到的是另一种情况：

- 电脑前已经有一条会话在跑
- 里面可能是 Claude Code、部署脚本、迁移、抓取或批处理
- 你离开了工位
- 你不想重开一条新 shell
- 你只想继续看原会话，并在必要时补一个轻量操作

TermPilot 的价值就在这里。它不让你重新组织上下文，而是让原来的上下文继续可达。

## 2. 它更适合长期任务

如果一个任务会持续运行一段时间，你真正关心的往往不是“远程进入机器”，而是：

- 它现在还在不在跑
- 最新输出是什么
- 我能不能补一条命令
- 我能不能发一个中断
- 我能不能把它停掉

这就是 TermPilot 当前最擅长的路径：

- 任务持续运行
- 人不一定一直坐在电脑前
- 手机主要负责查看和轻控制

## 3. 它把手机端收成了“任务面板”

TermPilot 当前不是在手机上复刻桌面终端，而是故意做了更窄的取舍：

- 先看会话列表
- 再进入当前会话
- 以查看输出为主
- 轻量输入、发快捷键、关闭会话

这让它更像：

- 长期任务的观察面板
- 共享终端会话的移动入口
- 桌面工作流的补充，而不是替代品

## 4. 它的部署路径很短

当前主路径本质上只有三端：

- 服务器：`termpilot relay`
- 电脑：`termpilot agent`
- 手机：打开 relay 地址

第一次配对完成后，日常最短路径通常就是：

```bash
termpilot agent
termpilot claude code
```

然后在手机上继续看同一条会话。

## 5. 它更符合“个人设备跨端继续”的模型

当前连接模型是：

- 电脑主动连 relay
- 手机主动连 relay
- relay 负责配对、scope 和加密信封转发
- 会话标题与终端输出保留在 agent 所在电脑

这意味着：

- 电脑自己不用暴露入站端口
- 手机不用进入复杂网络环境
- 公网部署更容易收敛到一个域名入口

## 6. 它不适合什么

TermPilot 当前并不打算解决这些问题：

- 远程桌面
- 图形界面控制
- 在手机上长时间重度终端编辑
- 自动接管任意已有终端标签页
- 通用多租户运维平台

它的价值来自聚焦，而不是覆盖所有远程场景。

## 7. 什么时候适合用它

如果你现在更接近下面这些场景，TermPilot 通常是合适的：

- 你经常有一条长期任务在电脑上持续运行
- 你离开桌面后，主要想继续看输出、补一条命令、发一个快捷控制
- 你不希望为了手机访问而把会话内容长期放进服务端
- 你想把公网入口收敛成一个 relay，而不是给电脑单独暴露更多入口

如果你真正需要的是远程桌面、图形化控制或手机上的重度编辑体验，TermPilot 就不是最合适的工具。

## 8. 一句话总结

TermPilot 的核心价值不是“远程进终端”。

它真正做的是：

**让一条已经在电脑上运行的终端会话，可以被手机继续接上，而且这件事足够轻、足够顺。**
