# TermPilot 架构方案

## 1. 产品目标

TermPilot 要解决的是一个明确场景：

- 电脑上正在运行终端里的智能体任务
- 用户离开电脑后，任务继续运行
- 用户可以在手机上查看同一个会话的持续输出
- 用户可以在手机上继续发命令、关闭会话、切换会话

它解决的是“终端会话跨端延续”，不是远程桌面。

## 2. 核心边界

第一版只管理 TermPilot 体系内的会话。

这意味着：

- 通过 TermPilot 创建的会话，电脑和手机都应该能看到
- 历史上已经存在、但不属于 TermPilot 的终端，不在第一版范围内
- 用户手动新开的普通 Terminal 或 iTerm 标签页，也不要求自动接管

一句话：

**只有进入统一会话池的会话，才要求手机和电脑双端互通。**

## 3. 总体拓扑

```text
手机端 PWA  --wss-->  relay  <--wss--  PC 端 agent
                           |
                    在线状态、消息转发、
                    输出缓冲、会话元数据
```

关键原则：

- 手机不开放端口
- PC 不开放端口
- 手机和 PC 都主动向 relay 发起连接
- 所有命令执行都发生在 PC 上

## 4. 会话底座

当前主方案固定为 `tmux`。

原因很直接：

- 同一个会话需要同时被电脑和手机查看
- 两边需要随时切换
- 长时间流式输出时，会话必须稳定存在

这些诉求和 `tmux` 的能力天然一致。

## 5. 三端职责

### 5.1 `agent/`

PC 端 agent 负责：

- 连接 relay
- 管理 `tmux` 会话
- 同步会话列表
- 接收远程输入、关闭等控制命令
- 持续读取输出并推送给 relay

当前实现技术：

- TypeScript
- Node.js
- `tmux`

### 5.2 `relay/`

relay 负责：

- 接收手机端和 agent 的 WebSocket 连接
- 设备认证
- 消息路由
- 最近输出缓冲
- 会话元数据存储

当前实现技术：

- TypeScript
- Node.js
- Fastify
- WebSocket
- PostgreSQL

说明：

- 设置 `DATABASE_URL` 时使用 PostgreSQL
- 本地开发未设置时，允许退回内存模式

### 5.3 `app/`

手机端负责：

- 展示设备状态和会话列表
- 查看当前会话输出
- 发送命令和快捷键
- 创建和关闭会话

当前实现技术：

- React
- Vite
- Tailwind CSS
- xterm.js
- PWA

## 6. 统一会话池

统一会话池是这个产品的核心概念。

它要求：

- 手机创建的会话，电脑端立刻可见
- 电脑创建的会话，手机端立刻可见
- 任一端关闭会话，另一端立刻感知
- 会话输出持续同步

## 7. 当前实现策略

当前版本采用的是一个务实策略：

- 输出同步先用 `tmux capture-pane` 快照替换模式
- relay 缓冲最近一段输出，供手机重连后补拉
- app 用 `xterm.js` 渲染当前终端内容

这不是最终形态，但它已经足够支撑“远程查看和控制同一个会话”的主目标。
