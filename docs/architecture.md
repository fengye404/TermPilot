# TermPilot 架构方案

## 1. 产品目标

TermPilot 要解决的是一个很具体的场景：

- 电脑上正在运行终端里的智能体任务，例如 Claude Code、OpenCode
- 用户需要离开电脑
- 任务继续在电脑上运行
- 用户可以在手机上持续查看流式输出，并在需要时继续发命令

这个产品的目标是“终端会话跨端延续”，不是“远程桌面”。

## 2. 当前已经明确的核心需求

结合这次讨论，产品必须满足下面几条：

- 同一个终端会话，电脑和手机可以同时查看。
- 当智能体持续流式输出时，电脑和手机两边都要实时同步显示。
- 在电脑上创建的会话，手机上必须能看到，也必须能关闭。
- 在手机上创建的会话，电脑上必须能看到，也必须能关闭。
- 两端看到的是同一个统一会话池，而不是各自维护一套独立会话。

这几条需求决定了，系统不能只是“手机远程开一个新终端”，而必须围绕“共享同一个真实会话”来设计。

## 3. 第一版边界

第一版要把边界定清楚：

- TermPilot 只管理“接入到它体系内”的会话。
- 历史上已经存在、但不是由 TermPilot 管理的终端会话，不在第一版范围内。
- 在 agent 启动之后，如果用户手动新开了一个普通 Terminal 或 iTerm 标签页，但不是通过 TermPilot 创建，这种会话也不要求自动同步。

换句话说：

**只有通过 TermPilot 入口创建，或者被 TermPilot 明确接管的会话，才要求手机和电脑双端互通。**

这条边界非常重要。它能让第一版可实现、可维护，也能避免后续为了“接管任意本地终端”而把复杂度拉爆。

## 4. 总体拓扑

```text
手机端  --wss-->  中继服务  <--wss--  PC 端 agent
                      |
               认证、路由、在线状态、
                最近输出缓冲、配对
```

关键原则：

- 手机不开放端口
- PC 不开放端口
- 手机和 PC 都主动向云端发起连接
- 所有命令执行都发生在 PC 上
- 中继服务只负责转发和轻量状态，不负责执行命令

## 5. 会话模型选择

这次讨论之后，核心结论已经比较明确：

- 主方案应当使用 `tmux`
- `node-pty` 可以作为技术验证手段，但不应作为长期主架构

### 5.1 为什么主方案要选 `tmux`

因为你的真实需求不是“远程新建一个终端”，而是：

- 电脑前和手机上能同时看同一个会话
- 两边随时切换
- 会话尽量长期存在
- 智能体任务长时间流式输出时，两边都能同步看到

这些诉求和 `tmux` 的能力几乎天然一致：

- 一个会话可以被多个客户端同时附着
- 本地终端和手机都可以连接到同一个会话
- agent 即使短暂重连，`tmux` 会话也更容易继续保活
- 多会话、多窗口、滚动历史，本来就是 `tmux` 的强项

### 5.2 为什么不把 `node-pty` 当主方案

如果用 `node-pty` 作为主架构，本质是：

- agent 自己创建一批独立 PTY 进程
- 手机看到的是 agent 自己养出来的终端

它适合快速做原型，但和你的核心诉求有冲突：

- 本地终端和手机不天然共享同一个真实会话
- agent 进程异常时，会话存活能力较弱
- 接入现有工作流时不够自然

因此，`node-pty` 最多只适合作为早期实验，不适合成为产品主线。

## 6. 建议的使用方式

为了让系统稳定、边界清晰，建议用户工作流从第一版开始就明确成下面这样：

1. 先启动 PC 端 agent
2. 通过 TermPilot 提供的入口创建一个会话
3. 这个会话的底层对应一个 `tmux` 会话或窗口
4. 在这个会话里运行 Claude Code、OpenCode 等任务
5. 在电脑前时，用本地终端附着到这个会话
6. 出门后，在手机上打开同一个会话继续查看和控制

这样一来，“在电脑上创建的会话”和“在手机上创建的会话”本质上都会落到同一个统一会话池里，自然可以互相看见和互相控制。

## 7. 三端职责

### 7.1 PC 端 agent

PC 端 agent 的职责：

- 建立并维持一条向中继服务的长期出站连接
- 管理 `tmux` 会话或窗口
- 维护 TermPilot 体系内的会话列表
- 接收来自手机的创建、输入、关闭、调整尺寸等指令
- 读取会话输出并持续推送到中继服务
- 处理断线重连，并尽量不影响底层会话存活
- 安全保存设备凭证

重要边界：

- agent 管理的是 TermPilot 体系内的会话
- agent 不承诺接管任意已有 Terminal 或 iTerm 窗口

### 7.2 中继服务

中继服务的职责：

- 接收手机和 PC 端的 WebSocket 连接
- 完成用户认证与设备认证
- 负责设备配对
- 按设备和会话路由消息
- 维护在线状态
- 缓冲最近一段输出，供手机重连后补拉

重要边界：

- 中继服务不是 shell 执行器
- 中继服务不应该理解具体命令语义
- 中继服务尽量保持轻量，避免变成复杂业务中心

### 7.3 手机端

手机端的职责：

- 展示设备状态和会话列表
- 进入某个会话后查看实时输出
- 发送输入
- 创建新会话
- 关闭会话
- 网络中断后重新附着并补最近输出

手机端应当偏向“监控和轻量控制”，而不是试图在手机上做完整开发工作流。

## 8. 统一会话池

系统里应当存在一个“统一会话池”概念。

这意味着：

- 所有受 TermPilot 管理的会话都能在电脑和手机两端看到
- 会话的创建来源不重要，重要的是它是否进入了统一会话池
- 手机创建的会话，电脑端应当立刻可见
- 电脑创建的会话，手机端应当立刻可见
- 任一端关闭会话，另一端都应同步感知状态变化

建议的数据模型至少包含：

- 用户
- 设备
- 手机客户端连接
- 终端会话

示例：

```json
{
  "sid": "s_123",
  "deviceId": "pc_main",
  "name": "claude-main",
  "backend": "tmux",
  "shell": "/bin/zsh",
  "cwd": "/Users/fengye/workspace/TermPilot",
  "status": "running",
  "startedAt": "2026-03-10T09:00:00Z",
  "lastSeq": 1042,
  "lastActivityAt": "2026-03-10T09:05:00Z"
}
```

## 9. 通信协议方向

建议使用 WebSocket 作为实时通道，控制消息使用统一的 JSON 包装。

示例：

```json
{
  "type": "session.create",
  "reqId": "r_001",
  "deviceId": "pc_main",
  "payload": {
    "cwd": "/Users/fengye/workspace",
    "shell": "/bin/zsh",
    "name": "claude-main"
  }
}
```

建议至少定义这些消息类型：

- `session.create`：创建会话
- `session.list`：拉取会话列表
- `session.attach`：附着到某个会话
- `session.input`：向会话写入输入
- `session.resize`：调整终端尺寸
- `session.kill`：关闭会话
- `session.output`：会话输出
- `session.exit`：会话退出
- `session.state`：会话状态变化
- `error`：错误返回

协议上的关键要求：

- 每个请求带 `reqId`
- 每段输出带递增的 `seq`
- 手机重连后可以按 `seq` 请求补拉
- 输出可以先按文本帧传输，后续再看是否需要二进制帧

## 10. 流式输出要求

你非常在意的一点是：

- 当智能体在终端里持续“啪啪啪”输出时
- 手机上必须也能同步看到这个流

因此系统必须满足：

- 输出要低延迟转发
- 中继服务要有短期缓冲
- 手机断线后重连不能直接丢掉最近上下文
- 会话切换时不能把输出流打断得很严重

第一版不必追求完整历史永久保存，但至少要保留最近一段输出用于补拉。

## 11. 安全模型

最低安全要求：

- 全链路使用 `wss://`
- 用户身份和设备身份分离
- 每台 PC 有自己的长期设备凭证
- 手机使用用户登录态
- 手机只有在配对成功后才能控制对应 PC

建议的配对流程：

1. 用户在手机端登录
2. PC 端 agent 启动进入配对模式
3. 中继服务生成短期配对码
4. 手机确认配对码
5. 中继服务向 PC 下发长期设备凭证

第一版可以暂缓：

- 端到端加密
- 多人共享控制
- 细粒度权限管理

## 12. 第一版最小可用范围

第一版建议只做下面这些能力：

- 单用户
- 单台 PC
- 多个 TermPilot 管理的会话
- 会话列表
- 创建会话
- 关闭会话
- 手机查看实时输出
- 手机发送输入
- 电脑本地附着同一个会话
- 网络重连后的最近输出补拉

第一版明确不做：

- 远程桌面
- 自动接管历史终端
- 自动接管体系外新开的普通终端标签页
- 文件传输
- 推送通知
- 团队协作

## 13. 风险点

最大的风险主要有四个：

- 试图在第一版就接管任意本地终端窗口
- 低估重连和输出补拉复杂度
- 让中继服务积累过多终端状态
- 手机端想做得过重，偏离“监控和轻量控制”定位

## 14. 当前结论

基于目前已经确认的需求，推荐结论如下：

- TermPilot 的主方案应当围绕 `tmux`
- 统一会话池是产品核心，不是附加能力
- 双端同步的前提是会话属于 TermPilot 管理体系
- 中继服务只做转发、缓冲、认证、配对
- 第一版先把“能稳定看同一个会话并控制它”做好，不要扩展到远程桌面或接管任意终端
