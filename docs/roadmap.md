# TermPilot 产品演进路线图

这份文档不是功能清单，而是未来一段时间内的产品与架构演进方向说明。

它回答 4 个问题：

1. TermPilot 长期到底要变成什么
2. 当前实现和目标状态之间还差什么
3. 哪些能力应该优先做，哪些应该延后
4. 端到端加密、零知识 relay、本地优先存储应该怎样落地

## 1. 长期目标

TermPilot 的长期目标不是成为远程桌面，也不是做通用服务器运维平台。

长期目标只有一条主线：

**让一条运行在电脑上的长期终端会话，可以被手机或其他个人设备继续安全地接入，而且 relay 只是中转站，不理解内容，也不持有长期状态。**

对应的目标产品形态：

- 用户的主数据默认保留在 agent 所在电脑
- relay 只负责连接建立、消息转发和最小必要的在线协调
- 会话内容、历史、输入、输出默认对 relay 不可见
- 多设备访问建立在端到端加密和设备级授权之上
- UI 保持“移动端查看 + 轻控制 + 同一会话连续性”的核心定位

## 2. 北极星原则

未来所有大改动都优先服从下面这些原则：

### 2.1 本地优先

- 会话状态、历史、索引、授权元数据优先存放在 agent 所在电脑
- 不因为“方便实现”把主数据长期放到 relay

### 2.2 relay 零知识

- relay 只做消息中转、连接协调和最小在线状态管理
- relay 不应默认保存明文输出、明文输入、明文会话元数据
- relay 最终不应依赖数据库来维持核心产品价值

### 2.3 端到端加密优先于功能堆叠

- 未来的多设备能力、会话回放、共享功能，都应该建立在 E2EE 基础上
- 不为了先做“方便的云能力”破坏加密边界

### 2.4 单用户 / 个人设备优先

- 第一阶段先把“一个人的多设备终端连续性”做到极致
- 多人协作、团队共享、托管控制台都不是当前主线

### 2.5 UI 为长期任务服务

- 设计重点始终是观察、确认、轻输入、快捷控制
- 不追求在手机上完全替代桌面终端

## 3. 当前状态与目标状态的差距

### 3.1 当前状态

基于当前代码，TermPilot 现在具备这些特点：

- relay 负责 HTTP / WebSocket、配对、授权、会话元数据和最近输出缓冲
- relay 可以运行在内存模式，也可以接 PostgreSQL
- agent 周期性抓取 `tmux capture-pane` 快照并同步到 relay
- client 通过 relay 获取会话列表、输出和状态
- 当前加密边界主要还是 TLS / WSS 传输层，不是应用层端到端加密

### 3.2 目标状态

未来希望演进到：

- relay 不保存会话输出缓冲
- relay 不保存明文会话元数据
- relay 不保存长期授权数据库，或只保存不可逆、短生命周期、最小必要状态
- agent 和 client 之间建立应用层 E2EE
- 历史、搜索、收藏、通知上下文等能力都从 agent 本地派生

### 3.3 最大缺口

从今天到目标状态，中间最大的架构缺口有 4 个：

1. 当前协议仍以 relay 可见明文消息为前提
2. 当前会话元数据和授权模型仍偏服务端中心化
3. 当前输出同步依赖 relay 最近缓冲
4. 当前配对流程还没有设备密钥体系和长期信任链

## 4. 安全与隐私路线

这是未来最重要的一条主线。

## 4.1 目标安全模型

长期目标安全模型如下：

```text
client  <--E2EE-->  agent
   \                 /
    \---- relay ----/
```

其中：

- relay 能转发密文，但不能解密内容
- agent 是主数据拥有者
- client 只是被授权的查看 / 控制设备
- 会话输入、输出、元信息、历史都默认只在 agent 和已授权 client 之间可读

## 4.2 relay 的最终角色

未来 relay 应只保留这些职责：

- 提供公网可访问连接点
- 协调 agent 与 client 的在线连接
- 转发端到端加密后的业务消息
- 提供极短生命周期的配对引导能力
- 在必要时提供匿名化或最小化的连接健康信息

relay 不应该长期承担：

- 明文输出缓冲
- 明文会话索引
- 明文审计日志
- 长期会话历史
- 主要状态数据库

## 4.3 E2EE 演进路径

### 阶段 A：建立设备密钥体系

先引入设备级长期密钥，而不是继续只依赖 token：

- 每个 agent 生成长期设备身份密钥
- 每个 client 设备也生成自己的身份密钥
- 配对码不再只是兑换访问 token，而是引导建立受信任设备关系
- 本地保存受信任设备列表

这一阶段的目标不是完全 E2EE，而是先把“谁是谁”这件事建立成密码学事实。

### 阶段 B：业务消息改成应用层加密

在连接已经通过 TLS 的基础上，再做应用层消息加密：

- `session.input`
- `session.output`
- `session.state`
- `session.exit`
- 会话元数据同步消息

都改为对 relay 不可读的密文消息。

这一阶段之后，relay 看到的应只是：

- 路由目标
- 消息长度
- 时间信息
- 最少必要的连接元信息

### 阶段 C：去除 relay 明文缓冲依赖

当前 replay 依赖 relay 的最近输出缓冲，这和零知识目标冲突。

未来改成：

- 输出缓冲只存在 agent 本地
- client 重连时向 agent 请求加密 replay
- relay 只做密文转发

### 阶段 D：最小化服务端授权状态

长期不建议 relay 持有长期授权数据库。

可以演进为：

- 短期配对会话只保留极短 TTL
- 长期信任关系由 agent 本地维护
- relay 只验证最小握手材料和会话有效期
- 需要撤销时，由 agent 主动广播信任变更

## 4.4 审计与隐私边界

未来审计要分层：

- relay 只记录最小连接级事件，且尽量不含业务明文
- agent 本地记录详细审计事件
- 用户可选择是否保留本地审计历史

如果保留审计，也应该优先是：

- 哪个设备在什么时候连过
- 哪个授权何时建立 / 撤销
- 哪个会话被创建 / 关闭

而不是记录全部明文输入内容。

## 5. 存储路线

## 5.1 长期目标：agent 本地为主

未来主要数据应默认保存在 agent 所在电脑：

- 会话索引
- 输出缓冲
- 历史
- 授权关系
- 设备信任链
- 收藏 / 置顶 / 最近访问等偏好

## 5.2 relay 存储策略

长期建议分两档：

### 默认模式：无数据库 relay

- 不要求 PostgreSQL
- relay 重启不影响长期主数据
- relay 只是中转层

### 兼容模式：最小状态 relay

只在必要时保留：

- 极短生命周期配对状态
- 节点健康状态
- 可选的匿名化连接指标

并且应明确标注为“兼容模式”，不是产品理想形态。

## 5.3 本地备份与恢复

当主数据落在 agent 本地后，需要同步设计：

- 本地状态目录结构版本化
- 安全导出 / 导入
- 信任设备迁移
- 本地数据加密存储

否则“数据留在本地”会变成“无法恢复”。

## 6. 协议与同步路线

## 6.1 从快照替换走向更细粒度同步

当前快照替换方案能工作，但有明显上限：

- 带宽效率不高
- 终端复杂界面容易抖动
- replay 粒度粗

未来协议建议这样演进：

### 短期

- 保留快照方案，但补足更稳定的尺寸 / 颜色 / 样式一致性
- 强化 session resize 策略
- 让不同端的渲染更一致

### 中期

- 引入增量帧或 patch 模式
- 允许 client 请求“从某个序号之后”的精细补拉
- 降低 relay / agent / client 之间的重复数据传输

### 长期

- 建立真正适合 E2EE 和 replay 的流式终端同步协议
- 把“路由”、“状态”、“密文载荷”解耦成不同消息层

## 6.2 会话模型演进

未来会话对象建议分成两层：

- 可公开路由层：最小必要标识
- 受保护业务层：名称、目录、状态细节、终端信息

这样更适合 E2EE 目标，也更利于协议瘦身。

## 7. 性能路线

## 7.1 agent

重点方向：

- 降低 `tmux capture-pane` 轮询成本
- 更高效的变化检测
- 更细粒度的输出同步
- 多会话并发时的 CPU 和 I/O 控制

建议阶段：

- 优先做会话活跃度感知轮询
- 非活跃会话降低采样频率
- 正在被查看的会话提高同步优先级

## 7.2 relay

如果 relay 最终只是中转层，性能目标应更明确：

- 长连接稳定
- 低内存占用
- 不因缓冲或数据库而膨胀
- 易于水平扩展

这意味着未来 relay 的优化重点应该是：

- 路由效率
- 连接恢复
- 回压处理
- 心跳和在线状态模型

## 7.3 client

重点方向：

- 终端渲染性能
- 大量输出下的 DOM / xterm 稳定性
- 移动端输入响应
- 前后台切换和重连体验

建议：

- 区分“观察模式”和“交互模式”
- 后台时降低无意义刷新
- 对超大输出做分层渲染或视窗优化

## 8. UI 设计路线

## 8.1 设计系统持续化

当前已经有设计系统文档，后续需要继续把它产品化，而不是只停在颜色层面。

建议长期形成：

- 颜色 token
- 字体层级
- 间距系统
- 状态语义色
- 终端专用组件规范
- 移动端控制面板规范

## 8.2 桌面 Web UI

未来桌面 Web UI 的方向：

- 更像控制台 / workspace，而不是普通表单页面
- 更强的会话摘要能力
- 更好的会话切换效率
- 更明确的在线 / 离线 / 重连状态

## 8.3 移动端 UI

移动端需要继续坚持：

- 单手可达
- 大部分高频动作一屏内可完成
- 终端查看优先
- 输入做轻量化，不强推复杂编辑

建议补强：

- 快捷键分组
- 常用命令模板
- 震动 / 通知反馈
- 横屏专注模式

## 8.4 设计非目标

不建议把 UI 做成：

- 通用云管理后台
- 多层复杂菜单系统
- 手机端重度 IDE
- 花哨动画驱动的概念展示页

## 9. 产品体验路线

## 9.1 首次接入

未来最该优化的是首次接入路径：

- relay 启动
- agent 初始化
- 配对
- 首个共享会话创建

目标是把这条路径做到：

- 更少解释成本
- 更少手动配置
- 更少失败分叉

## 9.2 日常使用

日常核心路径应始终围绕：

1. 打开 app
2. 看到关键会话
3. 查看最新输出
4. 发一个轻量操作
5. 离开

所有新增功能都应该问一句：

**它有没有让这条主路径更顺？**

## 9.3 通知与唤回

未来通知应该更智能，而不是更吵：

- 会话退出
- agent 离线
- 长时间无输出
- 指定关键词出现

但通知规则应尽量本地化、可配置、可静默。

## 10. 可靠性路线

重点建设方向：

- agent 崩溃后的恢复
- relay 重启后的快速重连
- client 前后台切换后的会话恢复
- 版本升级兼容
- 本地状态目录迁移

建议建立更明确的可靠性等级：

- 单会话恢复成功率
- 重连耗时
- 输出丢帧率
- UI 可用性

## 11. 里程碑建议

下面是一条更现实的演进顺序。

## 里程碑 1：把当前单用户体验做扎实

重点：

- 同步稳定性
- resize / 颜色 / replay 一致性
- 移动端操作体验
- 本地状态可靠性
- 更稳的 UI 和测试覆盖

目标：

- 当前产品可被长期使用
- 不再频繁出现终端错位、重连混乱、状态漂移

## 里程碑 2：设备身份与信任体系

重点：

- agent / client 长期身份密钥
- 配对升级为信任建立
- 撤销 / 更换设备流程

目标：

- 把“设备关系”建立成密码学层面的真实对象

## 里程碑 3：应用层 E2EE

重点：

- input / output / state 消息加密
- relay 不可读业务明文
- agent 本地承担 replay 和历史

目标：

- relay 成为真正零知识中转层

## 里程碑 4：零数据库 relay 优先

重点：

- 去除对 PostgreSQL 的核心依赖
- 把主状态迁回 agent 本地
- relay 只保留短期协调能力

目标：

- 用户可以运行一个真正轻量的 relay

## 里程碑 5：高级体验

重点：

- 智能通知
- 常用命令模板
- 多端体验细化
- 更细粒度同步协议

目标：

- 在不牺牲安全边界的前提下提升日常效率

## 12. 决策结论

如果要把方向说得足够清楚，可以浓缩成下面这几句：

- TermPilot 的未来主线是“本地优先 + relay 零知识 + 端到端加密”
- relay 长期应该只是中转层，而不是数据中心
- 主数据应尽量留在 agent 所在电脑
- 功能演进优先级应服从安全边界，而不是反过来
- UI、协议、性能、可靠性都应该围绕“同一条长期终端会话连续性”服务
